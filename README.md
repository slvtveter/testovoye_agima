# AI-ассистент команды

Мини-дашборд для тестового задания: веб-сервис, который принимает вопрос пользователя, отправляет его в LLM и показывает ответ с сохранением истории запросов.

## Стек

- Backend: FastAPI (Python)
- Хранилище истории: SQLite
- LLM: Google Gemini через `google-genai`
- Frontend: чистый HTML/CSS/JS без сборки

## Функционал

- Поле ввода вопроса и кнопка "Спросить AI"
- Ответ модели в отдельном блоке с подсветкой и кнопкой "Копировать"
- История последних 5 запросов с временем и названием использованной модели
- Обработка ошибок: пустой вопрос, исчерпанная квота, недоступность API — без падения сервиса

## Системный промпт

Модель работает не как обычный chat-бот, а как внутренний AI-ассистент команды разработки: отвечает по делу, без вступлений, структурирует ответ списками и даёт конкретику (код, команды, шаги) там, где это уместно. Промпт описан в `app/gemini_client.py`.

## Учет контекста предыдущих сообщений

Сервис ведёт общий контекст диалога, а не отвечает на каждый вопрос изолированно:

- Последние 10 сообщений (вопрос+ответ) передаются в модель как история диалога при каждом новом запросе.
- Как только число сообщений в контексте превышает 10, самые старые сообщения сжимаются в одну краткую сводку (summary) отдельным вызовом модели — она хранится в таблице `conversation_state` и подменяет собой полный текст старых сообщений.
- В каждый новый запрос модель получает: сводку (если есть) + последние до 10 сообщений + текущий вопрос. Это не даёт контексту расти бесконечно и экономит токены/квоту.

Логика реализована в `app/gemini_client.py` (`ask_ai`, `summarize_history`) и `app/main.py` (эндпоинт `/api/ask`).

## Fallback-цепочка моделей

Так как у бесплатных моделей в Google AI Studio ограниченная дневная квота (у каждой модели — своя, отдельная), запрос идёт по цепочке из пяти моделей, чтобы суммарный запас запросов в день был намного больше, чем у одной модели:

1. `gemini-2.5-flash` — основная модель
2. `gemini-3-flash` — запасная при исчерпании квоты основной
3. `gemini-3.5-flash` — ещё одна запасная с отдельным лимитом
4. `gemini-2.5-flash-lite` — облегчённая модель как дополнительный резерв
5. `gemini-3.1-flash-lite` — финальный резерв с самым большим дневным лимитом

Если модель в цепочке вернула ошибку (например, квота исчерпана), сервис автоматически пробует следующую. Только если все пять моделей недоступны, пользователь получает понятное сообщение об ошибке вместо падения сервиса.

## Структура проекта

```
app/
  main.py            точки входа FastAPI, эндпоинты /, /api/ask, /api/history
  database.py        работа с SQLite (сохранение и чтение истории)
  gemini_client.py    вызов Gemini с fallback-цепочкой и системным промптом
  static/
    style.css         стили интерфейса
    app.js            логика фронтенда (запросы, отрисовка истории, копирование)
  templates/
    index.html        разметка страницы
requirements.txt
```

## Запуск локально

```
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

Создать файл `.env` в корне проекта с переменной:

```
GOOGLE_API_KEY=ваш_ключ_от_google_ai_studio
```

Запустить сервер:

```
uvicorn app.main:app --port 8000
```

Открыть в браузере `http://localhost:8000`.

## API

- `POST /api/ask` — принимает `{"question": "..."}`, возвращает сохранённую запись с ответом модели
- `GET /api/history` — возвращает последние 5 запросов

## Тесты

```
pytest
```

Покрыты ключевые сценарии без обращения к реальному Gemini API (вызовы модели замоканы, тесты не тратят квоту и не требуют интернета):

- `tests/test_database.py` — сохранение запроса, выборка последних 5 в правильном порядке, чтение/обновление состояния диалога для контекста
- `tests/test_gemini_client.py` — fallback на следующую модель при ошибке первой, исключение, если все модели в цепочке недоступны, поведение суммаризации при пустой истории
- `tests/test_api.py` — валидация пустого вопроса и превышения лимита символов, успешный сценарий `/api/ask` с сохранением в историю, корректный ответ при исчерпании всех моделей (503, а не падение сервиса)
